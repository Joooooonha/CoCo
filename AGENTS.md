# CoCo Agent Instructions

이 문서는 CoCo 프로젝트에서 AI 에이전트가 일하는 방법을 정의한다. 제품 범위와 완료 조건은 `SPEC.md`, 작업 방식은 이 문서를 기준으로 한다.

## 1. 지침 우선순위

1. 사용자의 최신 명시적 결정
2. `SPEC.md`의 확정 범위와 완료 조건
3. 이 문서의 개발 및 검증 규칙
4. 기존 코드베이스의 패턴

사용자의 결정으로 범위가 바뀌면 코드보다 `SPEC.md`를 먼저 또는 같은 단계에서 수정한다. `SPEC.md`의 스코프 제외 항목은 사용자가 문서 변경을 승인하기 전까지 구현하지 않는다.

`SPEC.md`에는 최소한 개발 계획, 완성 기준, 확인 시나리오, 스코프 제외가 있어야 한다. 이 중 하나가 없거나 서로 충돌하면 기능 개발 전에 문서를 보완한다.

## 2. Apple HIG 스킬 사용

### 적용 시점

SwiftUI 화면, 컴포넌트, 내비게이션, 상호작용 또는 시각 상태를 생성·수정·리뷰하기 전에 다음 로컬 스킬을 사용한다.

`HIGAgentSkills-main/SKILL.md`

이 폴더는 로컬 외부 의존성이며 Git에는 포함하지 않는다. 폴더가 없으면 UI 작업을 시작하기 전에 [justinwetch/HIGAgentSkills](https://github.com/justinwetch/HIGAgentSkills.git)에서 설치하고 `SKILL.md` 경로를 확인한다.

단순 모델, 서버 또는 데이터베이스 작업만 하는 경우에는 HIG 전체 로딩을 생략할 수 있다. 해당 작업이 사용자 화면 상태나 문구에 영향을 주면 HIG를 적용한다.

### 필수 로딩 절차

화면 작업마다 `SKILL.md`의 Loading Protocol을 그대로 따른다.

1. 플랫폼, 화면, 컴포넌트, 프레임워크와 작업 유형을 파악한다.
2. `routing-index.md`의 Tier 1 문서 16개를 모두 읽는다.
3. Tier 2에서 `designing-for-ios`를 읽는다.
4. 작업 키워드와 일치하는 Tier 3 문서를 모두 읽는다.
5. 선택한 Tier 2·3 문서의 `related` 문서를 한 단계 확장해 읽는다.
6. Tier 4는 직접 키워드가 일치하거나 `related`로 연결될 때만 읽는다.
7. 읽은 기준을 구현과 리뷰에 적용한다.

지도 탐색 화면은 최소한 `maps`, `sheets`, `lists-and-tables`, `buttons`, `feedback`, `gestures`, `modality` 관련 문서를 키워드 검사 대상으로 삼는다. 입력 화면은 `entering-data`, `text-fields`, `pickers`, 오류·진행 상태 관련 문서도 검사한다.

### 적용 원칙

- 정확한 치수나 동작을 말할 때는 읽은 HIG 문서의 근거를 사용하고 값을 임의로 만들지 않는다.
- HIG의 OS 27은 디자인 참고 버전이다. 배포 타깃 iOS 26.2에서 지원되지 않는 API는 사용하지 않는다.
- Figma는 시각적 의도와 콘텐츠 구조의 기준으로 사용한다.
- HIG는 iOS 상호작용, 접근성, 시스템 동작과 플랫폼 관습의 기준으로 사용한다.
- Figma와 HIG가 충돌하면 조용히 한쪽을 무시하지 않는다. 차이와 권장안을 사용자에게 알리고 결정받는다.
- 화면 단계의 `RESULT.md` 기록에는 주요 판단과 실제로 읽은 HIG 문서 목록을 남긴다.

Figma 디자인을 SwiftUI로 옮기는 작업에서는 사용 가능한 Figma MCP와 SwiftUI 관련 Figma 스킬을 함께 사용한다. 디자인을 보지 못했거나 노드가 불명확한 상태에서 화면을 임의로 완성하지 않는다.

## 3. iOS 개발 규칙

- SwiftUI를 우선한다.
- UIKit 브리징과 외부 패키지는 `SPEC.md`에 필요성이 추가되고 사용자가 승인한 경우에만 도입한다.
- 배포 타깃은 iOS 26.2이며, 프로젝트 설정을 바꾸기 전에 실기기 OS를 확인한다.
- 지원 범위 안에서 최신 SwiftUI와 MapKit API를 사용하고 deprecated API를 새로 추가하지 않는다.
- 상태 공유에는 Observation의 `@Observable`, 화면 로컬 상태에는 `@State`를 기본으로 한다.
- 네트워크 요청은 Swift Concurrency와 `URLSession`을 기본으로 한다.
- 인증 토큰은 UserDefaults나 소스 코드가 아니라 Keychain에 저장한다.
- 현재 위치 권한, Core Location 수집과 백그라운드 위치 기능을 추가하지 않는다.

### MapKit

- 지도 렌더링 코드는 `MapCanvasView` 경계에 집중시킨다.
- 공용 도메인 모델에는 `CLLocationCoordinate2D`를 저장하지 않는다.
- 변환은 MapKit 경계에서 수행한다.
- 코스 선은 `MapPolyline`, 요소는 `Annotation` 또는 적합한 네이티브 API로 표현한다.
- 하프시트가 축소된 상태에서 지도 상호작용이 가능해야 한다.
- 계획 경로는 `MKDirections.Request`의 `.walking`을 사용한다.
- 러닝 코스에 Kakao 자동차 길찾기 결과를 사용하지 않는다.
- Kakao 도보 길찾기는 제휴 승인이 확인되기 전까지 구현하지 않는다.

### UI 품질

- Dynamic Type에서 텍스트가 잘리거나 다른 콘텐츠를 가리지 않아야 한다.
- 아이콘 버튼에는 접근성 레이블과 필요한 툴팁 또는 힌트를 제공한다.
- 터치 영역, 대비, 시스템 제스처, Reduce Motion과 VoiceOver를 고려한다.
- 로딩·오류·빈 상태를 정상 상태와 같은 수준의 화면 상태로 구현한다.
- 화면 작업을 마치면 지원하는 실기기 크기와 최소 한 개의 작은 시뮬레이터에서 겹침과 잘림을 확인한다.

## 4. 서버 개발 규칙

- Java 21과 Spring Boot 4.1.x를 기준으로 한다.
- API, 도메인, 영속성 책임을 분리하되 현재 규모보다 깊은 계층을 미리 만들지 않는다.
- 입력은 Bean Validation으로 검증하고 안정적인 오류 코드가 있는 공통 오류 응답을 사용한다.
- 인증이 필요한 쓰기 작업은 서버에서 현재 사용자를 결정한다. 클라이언트가 보낸 `ownerId`를 신뢰하지 않는다.
- 코스 요소 수정·삭제는 코스 소유자만 가능하도록 서비스와 테스트에서 검증한다.
- Flyway 마이그레이션 없이 운영 스키마를 직접 변경하지 않는다.
- 엔티티를 API 응답으로 직접 노출하지 않는다.
- 목록 조회의 N+1, 무제한 결과와 반복 쿼리를 확인한다.

### 테스트

- 서비스 규칙은 단위 테스트 또는 슬라이스 테스트로 검증한다.
- Repository와 마이그레이션은 실제 PostgreSQL 기반 통합 테스트를 우선한다.
- 인증, 소유권, 스크랩 중복, 반응 중복과 잘못된 열거형 입력을 반드시 테스트한다.
- API 변경 시 성공 응답과 대표 오류 응답을 함께 검증한다.

## 5. 인프라와 비밀값

- Mac mini 배포는 Docker Compose를 기준으로 한다.
- PostgreSQL 포트를 공용 네트워크에 노출하지 않는다.
- iPhone과 서버의 초기 연결은 Tailscale 사설 네트워크와 HTTPS를 사용한다.
- 비밀번호, Kakao 키, 토큰, 인증서와 실제 운영 환경 파일을 Git에 커밋하지 않는다.
- 저장소에는 `.env.example`처럼 키 이름과 설명만 제공한다.
- 로그에 인증 토큰, 비밀번호 또는 전체 개인정보를 남기지 않는다.
- 배포 단계에서는 헬스 체크, 재시작 정책, 데이터 볼륨과 백업 복구 방법을 확인한다.

## 6. 단계별 진행과 확인

`SPEC.md`의 Phase 순서를 따른다. 한 번에 하나의 검증 가능한 마일스톤만 진행하고 실패한 검증을 남긴 채 다음 마일스톤으로 넘어가지 않는다.

- SwiftUI 학습 단계: 보통 5~15분 크기
- 서버와 데이터베이스 단계: 보통 20~40분 크기
- 배포 단계: 하나의 배포 또는 검증 목표 단위

승인된 마일스톤 안의 파일 읽기, 빌드, 테스트와 작은 수정은 연속해서 수행할 수 있다. 다음 경우에는 사용자 확인을 받는다.

- 화면의 구조나 Figma 해석을 결정할 때
- `SPEC.md` 범위 또는 데이터 계약을 바꿀 때
- 외부 패키지, UIKit 브리징 또는 새 외부 서비스를 도입할 때
- 실제 기기에서만 확인할 수 있는 단계가 끝났을 때
- 비밀값, 계정 설정 또는 외부 시스템 권한이 필요할 때

각 마일스톤 종료 시 다음을 전달한다.

- 무엇을 만들었는지
- 왜 이 방식을 사용했는지
- 빌드와 테스트 결과
- 사용자가 직접 확인할 한 가지 핵심 동작
- 새로 등장한 학습 키워드 1~2개

## 7. 기록

개발 첫 단계에서 프로젝트 루트에 `RESULT.md`와 `STUDY.md`를 만든다. 기존 기록을 지우지 않고 날짜와 단계 기준으로 아래에 추가한다.

### RESULT.md

- 구현 내용과 변경 파일
- 주요 의사결정과 근거
- 읽은 HIG 문서 목록
- 실행한 빌드·테스트와 결과
- `SPEC.md`와 달라진 부분 및 이유

### STUDY.md

- 사용자가 질문한 Swift, iOS, Spring 또는 인프라 개념과 짧은 답
- 단계별 학습 키워드 1~2개

문서만 개정하는 단계에서는 개발 기록 파일을 미리 만들지 않는다.

## 8. 변경 및 완료 기준

- 관련 없는 파일이나 사용자의 기존 변경을 되돌리지 않는다.
- 자동 생성 파일을 수동 편집하지 않는다.
- 기능 변경에는 위험도에 맞는 테스트를 추가한다.
- 코드와 `SPEC.md`가 다르면 완료로 처리하지 않는다.
- 완료 시 `SPEC.md`의 확인 시나리오와 완성 기준을 기준으로 디자인, 기능, 테스트를 각각 `달성`, `부분 달성`, `미달성`과 근거로 평가한다.
- 계획과 실제 구현이 달라진 지점은 `RESULT.md`의 최종 평가에 남긴다.
